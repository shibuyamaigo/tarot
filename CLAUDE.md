## 🏗 プロジェクト全体アーキテクチャ

このプロジェクトは2つのレイヤーで構成される。

### Layer A: ゲーム本体（最重要・先に作る）
- **技術**: HTML5 Canvas + Vanilla JavaScript
- **配置**: `/game/` ディレクトリ
- **デプロイ先**: GitHub Pages または Vercel Static
- **目標**: 1MB以下、60fps、Bluetooth直結

### Layer B: マーケサイト（後で・暫定）
- **技術**: Next.js 15 (App Router) + TypeScript + Tailwind
- **配置**: `/marketing/` ディレクトリ
- **デプロイ先**: Vercel
- **目標**: LP、ブログ、課金導線
- **現状**: 暫定で最小限のみ（CEOがすぐデプロイしたい）

**重要**: ゲーム本体は絶対にNext.js化しない。
理由は後述の「過去の失敗」セクション参照。

## ⚠️ 絶対にやってはいけないこと

### ゲーム本体について
1. **React / Next.js / Vueを導入しない** - 過去に試して品質が下がった
2. **画像ファイルを安易に増やさない** - 数式描画を優先
3. **重いライブラリを追加しない** - バンドルサイズ最優先
4. **3Dレンダリング（Three.js / WebGL）を提案しない** - レトロ感が崩壊
5. **実GPSを使わない** - 物理エンジンで擬似計算
6. **TypeScriptを強制しない** - シンプル優先（v1.0以降に検討）

### プロジェクト全体について
7. **チート対策を忘れない** - 全データ入力点でバリデーション
8. **物理計算式を勝手に変更しない** - DECISIONS.md参照
9. **コアアイデアを安易にコメントに書かない** - パクられ防止
10. **大規模リファクタを勝手に提案しない** - 動いているコードを尊重
11. **APIや実装状況をハルシネーションしない** - 不明なら明示すること

## 🎮 過去の失敗から学んだこと（最重要）

### 失敗L-001: Next.jsでゲーム本体を作ろうとした
**問題**:
- Canvasゲームループ（60fps）とReact再レンダリングが競合
- ハイドレーション待ちで初期描画遅延
- useEffectのライフサイクルが物理計算と合わない
- バンドルサイズが肥大化（数十KB→数百KB）

**学び**:
- ゲーム系はVanilla JS + Canvasが正解
- 「モダン = 正しい」とは限らない
- 用途に応じた技術選択が最重要

**対策**:
- ゲーム本体は絶対にVanilla JSで実装する
- Next.jsはマーケサイトのみに留める
- フレームワーク提案を受けても、Canvas用途では却下

詳細は `LEARNINGS.md` 参照。

### 失敗L-002: データ構造追加時の初期化不整合
**問題**:
- 新しいキャラクターstate（verytired, downhill）を追加
- characterStates配列は更新したが、images.characterオブジェクトを更新し忘れ
- 配列pushでTypeError発生、全キャラクター表示停止

**根本原因**:
```javascript
// 更新された定義
const characterStates = ['normal', 'tired', 'fight', 'verytired', 'downhill'];

// 更新し忘れたデータ構造
const images = {
  character: {
    normal: [], fight: [], tired: []
    // ← verytired, downhillが未定義
  }
};
```

**学び**:
- enum/state追加時は対応するデータ構造を同時更新必須
- 配列操作前の存在確認が重要
- エラーハンドリングがあっても根本解決にならない

**対策**:
- データ構造定義とstate定義を同一ファイルに配置
- 初期化時のバリデーション関数追加
- コード変更後の即座の動作確認を必須とする

詳細は `BUGFIX-LOG-v2.8.md` 参照。

### 失敗L-003: 物理計算パラメータの固定値問題
**問題**:
- 体重スライダーをUIに追加したがゲーム物理に反映されない
- `estimateCyclingPowerWatts` / `solveCyclingSpeedKph` で `totalMass = 82` 固定
- ゲームバランス完全破綻（40kgでも90kgでも同じパワーで登坂）

**根本原因**:
```javascript
// UI側は体重連動実装済み
state.debug.weight = Number(event.target.value);

// 物理計算側は固定値のまま
function estimateCyclingPowerWatts(speedKph, gradePercent) {
  const totalMass = 82; // ← 固定！
}
```

**学び**:
- UI変更時は対応するロジック層も必ず確認
- 物理計算の引数は全て外部から注入可能にする
- ゲームバランスに関わる変更は必ず実際にプレイテスト

**対策**:
- 物理計算関数の引数に体重パラメータ追加
- 設定値の外部化（PHYSICS_CONFIG）検討
- ゲームバランス変更時の必須テスト項目追加

詳細は `CRITICAL-BUGFIX-v2.8.2.md` 参照。

### 失敗L-004: Bluetooth実データ統合でのゲーム全体停止
**問題**:
- Bluetooth接続は成功（"接続まで成功"）
- HRダミーデータを実データに切り替え時にゲーム全体フリーズ
- 復旧不可、リバート必須となった

**推定原因**:
- ダミーデータ生成ロジックの性急な削除
- 実データコールバックの未実装・エラー
- データフロー断絶による undefined 参照エラー
- 60fpsゲームループでのBluetoothデータ同期問題

**学び**:
- Bluetooth統合は段階的実装必須（一度に切り替えない）
- ダミーシステムは実データ安定まで絶対保持
- 接続成功 ≠ データ受信成功
- リアルタイムゲームでの外部データ統合は高リスク

**対策**:
- Phase分割アプローチ：接続→表示→ロジック統合
- ダミーデータとリアルデータの共存設計
- フォールバック機能の必須実装
- 各段階での動作確認とバックアップ

詳細は `BLUETOOTH-INTEGRATION-PLAN.md` 参照。

### 失敗L-005: TEST MODEによるリアルデータ無視問題
**問題**:
- Bluetoothから実パワーデータ（45-46W）受信中だがゲームに反映されない
- TEST MODE（AUTO/MANUAL）がダミーデータを強制生成
- リアルデータ統合にREAL MODEが不足

**根本原因**:
```javascript
// MANUAL/AUTOモードでダミーデータを強制生成
if (s.debug.manual) {
  targetPower = estimateCyclingPowerWatts(targetRealSpeed, s.grade, s.debug.weight || 70);
  // ← 実Bluetoothデータを無視
} else {
  // AUTOモードでもダミーデータ生成
}
```

**学び**:
- Bluetoothデータ受信 ≠ ゲーム統合
- TEST MODEが開発用からプロダクト機能に変化
- リアルセンサーモードの明示的実装が必須

**対策**:
- REAL MODEを第3の選択肢として追加
- 実データ優先度の明確化
- Bluetooth接続状態に応じた自動モード切替
